# 시스템 개요: GitHub 기반 "LLM Wiki" 에이전트

이 문서는 다른 AI 도구(예: Google AI/Gemini)에게 넘겨서 "지금 구성된 것과
동일한 동작을 그쪽에서도 재현할 수 있는지" 판단하는 데 쓰기 위한 기술 요약이다.
Claude 브랜드에 종속된 서술은 최대한 배제하고, 구조·파일·동작 메커니즘을
있는 그대로 기술한다.

## 1. 목적

원본 자료(뉴스 요약, 차트 캡처, 계좌 현황, 개인 프로필 등)를 넣으면 LLM이
읽어서 마크다운 지식베이스("위키")를 만들고 갱신하며, 나중에 질문하면 그
위키를 근거로 답한다. 매번 원본을 처음부터 다시 분석하지 않고, 이미 정리된
노트를 계속 누적·재사용하는 것이 핵심이다. (원 아이디어 출처:
Andrej Karpathy, "LLM Wiki" 패턴,
https://gist.github.com/karpathy/442a6bf555914893e9891c11519de94f)

## 2. 저장소 구조

```
Repo_name/
├── CLAUDE.md                 ← 스키마 파일 (구조·규칙·워크플로우 정의, 아래 4번 참고)
├── README.md                  ← 사람이 보는 프로젝트 소개 + 사용법 요약
├── USAGE.md                   ← 초보자용 상세 사용설명서 (한글)
├── sources/                    ← 원본, 불변 입력 자료
│   ├── README.md
│   ├── test-memo-2026-07-13.md
│   ├── sk-hynix-rebound-analysis-2026-07-13.md
│   ├── sk-hynix-krw-chart-2026-07-13.png
│   ├── sk-hynix-adr-chart-2026-07-13.png
│   ├── portfolio-holdings-2026-07-13.md
│   └── user-profile-2026-07-13.md
├── wiki/                       ← LLM이 유지·관리하는 지식베이스
│   ├── index.md                ← 전체 페이지 카탈로그 (폴더별 1줄 요약)
│   ├── log.md                  ← 추가만 되는(append-only) 이벤트 로그
│   ├── summaries/               ← 소스 1건당 1페이지, 압축 요약
│   │   ├── test-memo.md
│   │   ├── sk-hynix-rebound-analysis-2026-07-13.md
│   │   ├── sk-hynix-adr-krw-chart-comparison-2026-07-13.md
│   │   └── portfolio-holdings-2026-07-13.md
│   ├── entities/                 ← 반복 등장하는 사람/조직/상품/사물 1개당 1페이지
│   │   ├── sk-hynix.md
│   │   ├── my-portfolio.md
│   │   └── user-profile.md
│   └── concepts/                 ← 반복 등장하는 개념/주제 1개당 1페이지 (아직 없음)
└── .claude/commands/            ← Claude Code 전용 슬래시 커맨드 정의 (5번 참고)
    ├── ingest.md
    ├── query.md
    └── lint.md
```

## 3. 각 계층의 역할

| 계층 | 성격 | 예시 |
| --- | --- | --- |
| `sources/` | 원본 그대로, 절대 수정/삭제하지 않음 | 뉴스 요약 텍스트, 증권 앱 차트 캡처(png), 계좌 테이블, 사용자 정보 |
| `wiki/summaries/` | 소스 1건을 압축한 노트 (소스와 1:1 대응에 가까움) | `sk-hynix-rebound-analysis-2026-07-13.md` |
| `wiki/entities/` | 여러 소스에 걸쳐 누적되는 "사물"에 대한 지속 갱신 페이지 | `sk-hynix.md`(종목), `my-portfolio.md`(포트폴리오), `user-profile.md`(사용자) |
| `wiki/concepts/` | 여러 곳에서 반복되는 개념/주제 (아직 미사용) | (예정) |
| `wiki/index.md` | 전체 페이지를 폴더별로 나열한 카탈로그, 검색 진입점 | — |
| `wiki/log.md` | 언제 무엇을 넣었고(`INGEST`) 무엇을 물었는지(`QUERY`) 시간순 기록 | — |
| `CLAUDE.md` | 위 모든 규칙과 워크플로우를 정의하는 스키마 문서 | — |

## 4. 스키마(`CLAUDE.md`)가 정의하는 것

- **페이지 형식**: 파일명은 kebab-case, 모든 페이지는 frontmatter
  (`title`/`created`/`updated`/`tags`)로 시작하고, 본문은 소스를 그대로 베끼지
  않고 "나중에 다시 읽고 싶은 압축된 글"로 쓰며, 끝에는 반드시 `## Sources`
  섹션으로 근거(원본 파일 또는 다른 위키 페이지)를 명시한다.
- **교차 링크**: 관련된 페이지끼리는 상대경로 마크다운 링크로 서로 연결하고,
  한쪽을 갱신하면 관련된 다른 페이지도 링크를 갱신하는지 확인한다.
- **`index.md` 규칙**: 폴더별로 묶어서 페이지당 한 줄(`- [제목](경로) — 한줄요약`).
- **`log.md` 규칙**: 추가만 하며, `INGEST`/`QUERY`/`LINT` 접두어로 파싱 가능한
  한 줄 기록을 남긴다.
- **3가지 워크플로우**:
  1. **Ingest** — 새 소스를 받으면: 기존 위키에서 관련 페이지를 찾아 우선
     업데이트하고(중복 페이지 생성 지양), 없으면 새로 만든다. 관련 교차링크를
     양방향으로 갱신하고, `index.md`/`log.md`를 갱신한다.
  2. **Query** — 질문을 받으면: `index.md`와 위키 전체를 검색해 관련 페이지를
     찾고, 어떤 페이지를 근거로 삼았는지 인용하며 답한다. 답변 중 위키에 없는
     새로운 유용한 정보가 나오면 "이걸 위키에 반영할지" 먼저 제안한다.
  3. **Lint** — 위키 전체를 훑어 모순, 더 이상 근거 없는 주장, 고아 페이지
     (index에서 도달 불가), 빠진 교차링크, 깨진 링크를 찾아 보고한다(자동
     수정하지 않음).

## 5. 지금 이 동작을 가능하게 하는 실행 메커니즘

지금은 **Claude Code**(Anthropic의 코딩 에이전트, 이 저장소를 파일시스템처럼
읽고 쓰고 git 명령을 실행할 수 있는 환경)에서 실행 중이다. 실제 동작 순서는:

1. 사용자가 채팅에 `/ingest <내용>`, `/query <질문>`, `/lint` 를 입력한다.
2. `.claude/commands/{ingest,query,lint}.md` 파일이 이걸 "CLAUDE.md의 워크플로우를
   그대로 따르라"는 지시문으로 확장한다 (일종의 매크로/프롬프트 템플릿).
3. LLM 에이전트가 `CLAUDE.md`, `wiki/index.md`, 관련 위키 페이지들을 실제로
   파일시스템에서 읽고, 파일을 새로 만들거나 수정한다.
4. 에이전트가 `git add / commit / push`를 직접 실행해 GitHub 원격 저장소
   (`stevekoo-ai/Repo_name`, 브랜치 `claude/ai-agent-impl-002tip`)에 반영한다.

즉 "위키를 관리하는 로직" 자체는 전부 **일반 텍스트 규칙(CLAUDE.md)** 에
있고, LLM은 그 규칙을 읽고 파일을 읽고 쓰는 것뿐이다 — 특정 벤더에 종속된
로직이 아니다. 종속적인 부분은 딱 두 가지: (a) 슬래시 커맨드라는 트리거 방식,
(b) 그 LLM이 실제로 파일을 읽고/쓰고 git을 실행할 수 있는 "도구 접근권한"을
가지고 있다는 점이다.

## 6. 다른 AI(Google AI/Gemini 등)로 이식하려면 필요한 것

핵심 판단 기준: **그 AI가 (a) 이 GitHub 저장소의 파일을 읽고 쓸 수 있는가,
(b) git commit/push를 실행(혹은 대행)할 수 있는가.**

- **가능한 경우** (예: 저장소에 연결된 코드 어시스턴트, IDE 확장, 또는 파일/저장소
  접근 권한이 있는 에이전트 모드): `CLAUDE.md` 파일을 그 도구가 참고하는
  "프로젝트 지침" 파일로 지정하고(도구마다 이름이 다를 수 있음 — 예:
  `GEMINI.md`, `AGENTS.md` 등, 혹은 세션 시작 시 시스템 프롬프트로 붙여넣기),
  `/ingest`, `/query`, `/lint`에 해당하는 지시문을 매번 프롬프트로 전달하면
  거의 동일하게 동작할 수 있다. 슬래시 커맨드 자동완성이 없으면 그냥
  `.claude/commands/ingest.md`에 적힌 지시문 전체를 매번 복사해 붙여넣는 것으로
  대체 가능하다.
- **불가능한 경우** (파일/저장소 접근권한이 없는 순수 채팅 인터페이스만 열려있는
  경우): 완전 자동화는 안 되고, 아래처럼 수동 절차로 대체해야 한다.
  1. 회사에서 Google AI 채팅창에 `CLAUDE.md` 내용 전체 + 관련 위키 페이지
     내용을 그대로 붙여넣는다.
  2. 새 소스(자료)를 붙여넣고 "위 규칙에 따라 이 소스를 위키에 반영해줘,
     새 페이지 전체 마크다운을 출력해줘"라고 요청한다.
  3. Google AI가 출력한 마크다운을 복사해서, 나중에(회사 밖에서 또는 GitHub
     웹 편집기로) 해당 파일에 붙여넣고 커밋한다.
  4. 즉 "판단과 글쓰기"는 회사에서 Google AI가 하고, "파일 반영과 git 커밋"은
     나중에 별도로 처리하는 2단계 워크플로우가 된다.

## 7. 현재 위키 상태 스냅샷 (참고용)

- 페이지 수: summaries 4개, entities 3개, concepts 0개
- 다루는 내용: SK하이닉스 주가 동향(본주 급락 및 ADR 동반 하락), 개인 포트폴리오
  현황(4개 계좌, SK하이닉스가 약 65% 비중), 사용자 프로필(SK하이닉스 재직,
  주거·청약 계획)
- 전체 목록은 `wiki/index.md`, 시간순 이력은 `wiki/log.md`에서 확인 가능

## 8. 요약

이 시스템은 특정 AI 제품에 종속된 게 아니라 **"규칙 파일 + 파일을 읽고 쓸 수
있는 에이전트 + git"** 조합이면 어디서든 재현 가능한 패턴이다. Google AI 쪽에서
검토할 때 확인해야 할 것은 결국 "그 AI가 이 저장소의 파일을 실제로 읽고 쓰고
커밋할 수 있는 권한/도구를 갖고 있는가" 하나로 좁혀진다.
